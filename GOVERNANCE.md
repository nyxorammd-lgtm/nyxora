# Governance

NYXORA follows a **lazy consensus** model.

## Decision Making

- **Minor changes** (bug fixes, docs, tests): committer can decide
- **Major changes** (new transports, architectural): discuss via GitHub Issues/Discussions
- **Breaking changes**: require consensus from maintainers and 2-week notice

## Voting

- Majority vote among maintainers
- Maintainers can veto on security or stability grounds
- 2 business days minimum for voting period

## RFC Process

For significant changes:
1. Open a [Discussion](https://github.com/nyxorammd-lgtm/nyxora/discussions) with `[RFC]` prefix
2. Describe motivation, design, and alternatives
3. Minimum 1 week for community feedback
4. Maintainers vote to accept, reject, or request changes
5. Accepted RFCs become issues with `rfc/accepted` label

## Roles

- **User**: anyone using NYXORA
- **Contributor**: anyone submitting PRs/issues
- **Maintainer**: trusted contributor with merge access
- **Lead Maintainer**: final decision authority

## Code of Conduct

All participants must follow our [Code of Conduct](CODE_OF_CONDUCT.md).
